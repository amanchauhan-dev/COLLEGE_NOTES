## Message Authentication Codes (MAC)

## 6.1 Message Authentication Codes (MAC)

=> Message authentication is a service used to verify the integrity and authenticity of a message.

=> Message integrity ensures that the message has not been altered during transit.

=> Message authentication guarantees that the sender is who they claim to be.

=> A MAC algorithm, or keyed hash function, takes a secret key and an arbitrary-length message as input.

=> The output is a fixed-length MAC value that protects the message's data integrity and authenticity.

=> Verifiers use the MAC to detect any changes made to the message content.

=> MACs are typically based on block ciphers or hash functions.

=> HMAC is a common implementation used in protocols like Transport Layer Security (TLS).

### Properties of Message Authentication Codes

1. `Cryptographic checksum`: Generates a secure authentication tag for a message.
2. `Symmetric`: Uses a shared secret key between the signing and verifying parties.
3. `Arbitrary message size`: Can accept messages of any length.
4. `Fixed output length`: Produces a fixed-size tag regardless of input size.
5. `Message integrity`: Detects any manipulation of the message during transit.
6. `Message authentication`: Assures the receiver of the message's origin.
7. `No non-repudiation`: Since keys are shared, it cannot prove which of the two parties created the message.

### Applications of MAC

1. Broadcast scenarios where the same message is sent to multiple destinations.
2. Authentication of computer programs in plaintext.
3. High-load exchanges where the receiver cannot afford the time to decrypt all incoming messages.

### MAC Mathematical Representation

=> The MAC is generated by a function $C$.

=> Formula: $MAC = C(K, M)$.

=> $M$ represents the variable-length message.

=> $K$ represents the secret key shared only by the sender and receiver.

=> $C(K, M)$ results in a fixed-length authenticator.

### Security of MAC

=> Security generally depends on the bit length of the key.

=> Brute force attacks are the primary weakness if key lengths are short.

=> In a ciphertext-only attack, an opponent performs $P_i = D(K_i, C)$ for all possible keys until an acceptable plaintext is found.

### Brute Force Search (Key size > MAC size)

=> **Round 1**: Given $M_1$ and $MAC_1 = C(K_1, M_1)$, the opponent computes $MAC_i = C(K_i, M_1)$ for all $2^k$ keys.

=> The number of matches in Round 1 is approximately $2^{(k - n)}$.

=> **Round 2**: Given $M_2$ and $MAC_2 = C(K, M_2)$, the opponent computes $MAC_i$ for the remaining $2^{(k - n)}$ keys.

=> The number of matches in Round 2 is approximately $2^{(k - 2 \times n)}$.

=> On average, $\alpha$ rounds are needed if $K = \alpha \times n$.

#### Brute Force Search (Key size <= MAC size)

=> The first round will likely produce a single match.

=> If more than one match occurs, the opponent tests the keys against a new (message, MAC) pair.

### MAC Requirements for Function C

1. Computational Infeasibility: If an opponent knows $M$ and $C(K, M)$, it should be impossible to construct a new message $M'$ such that $C(K, M') = C(K, M)$.
2. Uniform Distribution: For randomly chosen messages, the probability that $C(K, M) = C(K, M')$ should be $2^{-n}$ (where $n$ is the number of bits in the MAC).
3. Transformation Resistance: $C(K, M)$ should not be predictable even if $M'$ is a known transformation of $M$.

### 6.1.1 Authentication Requirements

=> Message authentication is a procedure to verify that messages are from the alleged source and remain unaltered.

=> Digital signatures differ by providing measures to counter source repudiation.

#### Types of Attacks

1. Disclosure: Releasing message contents to unauthorized persons.
2. Traffic analysis: Discovering patterns of traffic between parties.
3. Masquerade: Inserting messages into the network from a fraudulent source.
4. Sequence modification: Inserting, deleting, or reordering a sequence of messages.
5. Content modification: Changing the actual contents of a message.
6. Timing modification: Delaying or replaying messages.
7. Source repudiation: Denial of transmission by the sender.
8. Destination repudiation: Denial of receipt by the receiver.

### Message authentication code based on DES

=> This algorithm uses the Cipher Block Chaining (CBC) mode of DES.

=> It uses an initialization vector (IV) of zero.

=> The data is grouped into contiguous 64-bit blocks ($D_1, D_2, \dots, D_N$).

=> The Data Authentication Code (DAC) is calculated as follows:

=> $O_1 = E(K, D_1)$

=> $O_2 = E(K, [D_2 \oplus O_1])$

=> $O_3 = E(K, [D_3 \oplus O_2])$

=> $O_N = E(K, [D_N \oplus O_{N-1}])$

=> The final DAC consists of either the entire block $O_N$ or the leftmost $M$ bits of the block ($16 \leq M \leq 64$).

## 6.2 MAC Based on Hash Functions

=> The IPsec authentication scheme utilizes Hashed Message Authentication Codes (HMAC).

=> HMAC is defined as an encrypted message digest described in RFC 1024.

=> It uses a shared secret key between two parties instead of public key methods.

#### Objectives for HMAC

1. To use existing hash functions without any internal modifications.
2. To allow easy replacement of the embedded hash function if more secure or faster ones are discovered.
3. To handle and use keys in a simple, straightforward manner.
4. To preserve the original performance of the hash function without significant degradation.
5. To provide a well-understood cryptographic analysis of the strength of the mechanism based on the embedded hash.

#### HMAC algorithm

=> **$H$** = Embedded hash function.

=> **$IV$** = Initial value input to the hash function.

=> **$M$** = Message input to HMAC.

=> **$Y_i$** = $i^{th}$ block of message $M$, where $0 \le i \le (L - 1)$.

=> **$L$** = Total number of blocks in message $M$.

=> **$b$** = Number of bits in a block.

=> **$n$** = Length of the hash code produced by the embedded hash function.

=> **$K$** = Secret key (recommended length is $\ge n$).

=> **$K^{+}$** = Secret key $K$ padded with zeros on the left so the result is $b$ bits long.

=> **$ipad$** = Input padding; the value 00110110 (36 in hex) repeated $b/8$ times.

=> **$opad$** = Output padding; the value 01011100 (5C in hex) repeated $b/8$ times.

#### Steps in HMAC Calculation

1. Append zeros to the left end of $K$ to create a $b$-bit string $K^{+}$.
2. XOR $K^{+}$ with $ipad$ to produce the $b$-bit block $S_i$.
3. Append message $M$ to $S_i$.
4. Apply the hash function $H$ to the stream generated in the previous step.
5. XOR $K^{+}$ with $opad$ to produce the $b$-bit block $S_o$.
6. Append the hash result from step 4 to $S_o$.
7. Apply the hash function $H$ to the stream generated in step 6 and output the result.

#### HMAC Mathematical Expression

=> $HMAC(K, M) = H[(K^{+} \oplus opad) \ || \ H[(K^{+} \oplus ipad) \ || \ M]]$

#### Efficient Implementation of HMAC

=> A more efficient version precomputes two specific quantities.

=> Precomputed value 1: $f(IV, (K^{+} \oplus ipad))$.

=> Precomputed value 2: $f(IV, (K^{+} \oplus opad))$.

=> Here, $f$ represents the compression function for the hash function.

#### HMAC Security

=> The security of HMAC is directly related to the security of the underlying hash algorithm.

=> Attacking HMAC typically involves two primary methods:

1. Brute-force attack on the key: Complexity is in the order of $2^n$ where $n$ is the chaining variable bit-width.
2. Birthday attack: In the order of $2^{n/2}$ for a hash length of $n$.

#### HMAC vs MD5 Security

=> HMAC is significantly more secure than MD5 against birthday attacks.

=> In MD5, an attacker can choose any set of messages offline to find a collision $H(M) = H(M')$.

=> In HMAC, the attacker does not know the secret key $K$ and cannot generate messages offline.

=> For a 128-bit hash code, monitoring a 1 Gbps line would require approximately 250,000 years to observe enough blocks to be successful.

=> This level of difficulty makes the attack quite infeasible.

## 6.3 MACs Based on Block Ciphers

=> Cipher-based Message Authentication Code (CMAC) is an algorithm based on block ciphers.

=> It is commonly used with the AES and Triple DES (3DES) algorithms.

=> The message is divided into blocks of size equal to the block size of the underlying cipher (e.g., 128 bits for AES).

=> It essentially uses Cipher Block Chaining (CBC) encryption, retaining the result of the last block encryption as the MAC value.

=> To prevent specific classes of security attacks, the last block is processed with a "subkey" before the final encryption.

=> This subkey is denoted as $K1$ or $K2$.

=> The choice between $K1$ and $K2$ depends on whether the last message block required padding.

=> Only parties who know the original cipher key can compute these subkey values.

#### CMAC Calculation Logic

=> $C_1 = E(K, M_1)$

=> $C_2 = E(K, [M_2 \oplus C_1])$

=> $C_3 = E(K, [M_3 \oplus C_2])$

=> $C_n = E(K, [M_n \oplus C_{n-1} \oplus K_1])$

=> $T = MSB_{tlen}(C_n)$

=> $T$ represents the final Message Authentication Code.

=> $Tlen$ is the bit length of the tag $T$.

=> $MSBs(X)$ refers to selecting the $s$ leftmost bits of the resulting bit string $X$.

### 6.3.1 Data Authentication Algorithm

=> This algorithm is based on DES and has been a widely used MAC standard for many years.

=> It operates using the Cipher Block Chaining (CBC) mode of operation.

=> It utilizes an initialization vector (IV) of zero.

=> The input data is grouped into contiguous 64-bit blocks ($D_1, D_2, \dots, D_N$).

#### DAC Calculation Steps

=> $O_1 = E(K, D_1)$

=> $O_2 = E(K, [D_2 \oplus O_1])$

=> $O_3 = E(K, [D_3 \oplus O_2])$

=> $O_N = E(K, [D_N \oplus O_{N-1}])$

=> The resulting Data Authentication Code (DAC) can be the entire block $O_N$.

=> Alternatively, the DAC can be the leftmost $M$ bits of $O_N$, where $16 \le M \le 64$.
